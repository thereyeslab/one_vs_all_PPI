# Inputs and Outputs

## Overview

This document describes all major inputs, intermediate files, and outputs generated by the pooled-PPI workflow.

The workflow consists of:

```text
Protein filtering
        ↓
Pool generation
        ↓
AlphaFold3 input generation
        ↓
Input validation
        ↓
AlphaFold3 prediction
        ↓
Score extraction
        ↓
Size correction
        ↓
Replicate aggregation
```

Understanding these files is useful for:

* Quality control
* Troubleshooting
* Workflow customization
* Downstream analysis
* Reproducibility

---

# Directory Structure

A typical project output directory looks like:

```text
project_output/
└── project_name/
    ├── pools.tsv
    ├── pool_summary.tsv
    ├── pool_chain_mapping.tsv
    ├── af3_json_validation_summary.tsv
    ├── pair_scores_raw.tsv
    ├── prediction_summary.tsv
    ├── pair_scores_size_corrected_observations.tsv
    ├── pair_scores_size_corrected_aggregated.tsv
    ├── size_correction_model.tsv
    │
    └── alphafold3_jsons/
```

AlphaFold prediction outputs should be placed in:

```text
alphafold3_predictions/
├── pool_00001/
├── pool_00002/
├── pool_00003/
└── ...
```

# Phase 0 — Protein Filtering

## Inputs

### Proteome FASTA

Example:

```text
caulobacter_proteome.fasta
```

Contains all proteins from the organism of interest.

---

### UniProt Annotation Table

Example:

```text
caulobacter_uniprot_annotations.tsv
```

Used to identify proteins that should be excluded from the screening. 

#### Important
The required columns defined in the configs/proteome_filtering_config.yaml should exist in the uniprot annotation file. 
Example of uniprot query:
```text
curl -L "https://rest.uniprot.org/uniprotkb/stream?query=proteome:UP000001364&format=tsv&fields=accession,id,gene_names,protein_name,organism_name,length,cc_subcellular_location,ft_transmem,go,go_c,go_p,go_f,keyword,sequence" \
 -o data/raw/caulobacter_uniprot_annotations.tsv
```
The filtering module is a dynamic filtering method. So it supports your desired filtering. Just make sure the you configure it with
the correct format (docs/proteom_filtering.md)

---

## Outputs

### candidate_proteins.fasta

Contains all proteins retained after filtering.

This is the primary input for pooled-PPI.

---

### candidate_proteins.csv

Tabular representation of retained proteins.

Contains:

* UniProt identifiers
* Gene names
* Protein annotations
* Metadata

---

### excluded_proteins_with_reasons.csv

Contains proteins removed during filtering together with exclusion reasons.

Examples:

```text
ribosomal
membrane_bound
periplasmic
outer_membrane_or_cell_wall_synthesis
```

---

### all_proteins_annotated.csv

Complete annotated protein table.

Useful for:

* Auditing filtering decisions
* Quality control
* Developing new filtering rules

# Phase 1 — Pool Generation

## Input

### candidate_proteins.fasta

Generated during filtering.

---

## Outputs

### pools.tsv

Primary pooling output.

Each row corresponds to one pool.

Columns:

| Column       | Description                     |
| ------------ | ------------------------------- |
| pool_id      | Unique pool identifier          |
| mode         | Pooling mode                    |
| replicate    | Replicate number                |
| target_id    | Bait protein                    |
| protein_ids  | All proteins in the pool        |
| prey_ids     | Prey proteins only              |
| n_proteins   | Total proteins in pool          |
| n_prey       | Number of prey proteins         |
| total_length | Total amino-acid length of pool |

Example:

```text
pool_id      pool_00001
target_id    RFA1
n_proteins   6
total_length 2979
```

---

### pool_summary.tsv

Pooling coverage summary.

Columns:

| Column         | Description                          |
| -------------- | ------------------------------------ |
| target_id      | Bait protein                         |
| prey_id        | Prey protein                         |
| prey_length    | Protein length                       |
| n_observations | Number of pools containing this prey |

Useful for verifying:

* Pool coverage
* Replicate coverage
* Pooling quality

---

# Phase 2 — AlphaFold3 Input Generation

## Inputs

### candidate_proteins.fasta

Protein sequences.

### pools.tsv

Pool assignments.

---

## Outputs

### alphafold3_jsons/

Contains one AlphaFold-compatible JSON file per pool.

Example:

```text
alphafold3_jsons/
├── pool_00001.json
├── pool_00002.json
├── pool_00003.json
└── ...
```

These files can be submitted to:

* AlphaFold Server
* AlphaFold3 HPC installation

depending on the selected dialect.

---

### Batch JSON Files (AlphaFold Server)

For AlphaFold Server workflows, pooled-PPI also generates batch files.

Example:

```text
batch_001.json
batch_002.json
```

Each batch contains up to 100 AlphaFold jobs.

These files can be uploaded directly to AlphaFold Server and greatly reduce manual work.

---

### pool_chain_mapping.tsv

One of the most important workflow files.

Maps AlphaFold chain identifiers back to biological proteins.

Columns:

| Column                    | Description                          |
| ------------------------- | ------------------------------------ |
| pool_name                 | Pool identifier                      |
| entity_index              | AlphaFold entity number              |
| pseudo_chain_id           | AlphaFold chain ID                   |
| protein_id                | Biological protein ID                |
| count                     | Number of copies                     |
| protein_length            | Protein length                       |
| total_residues_for_entity | Total residues represented by entity |

Example:

```text
pool_name     pseudo_chain_id   protein_id
pool_00001    A                 RFA1
pool_00001    B                 RAD27
pool_00001    C                 POL30
```

This file is required during score extraction.

## Outputs

| ------------ | ----------------- |
| json_file    | JSON filename     |

Predictions may be generated using:

## Output

Example:

```text
alphafold3_predictions/
```

## Important Files

### Summary Confidence Files

Example:

```text
pool_00001_seed-1_sample-0_summary_confidences.json
```

These files contain:

* pairwise ipTM scores
* ranking scores
* confidence metrics

They are the primary source used during score extraction.

Do not delete them.

| chain_b             | Chain B                 |

| protein_b_length    | Length of protein B     |
| pair_length_sum     | Combined protein length |
| chain_pair_iptm     | Pairwise ipTM           |
| chain_pair_pae_min  | Minimum interface PAE   |
| chain_pair_plddt    | Interface pLDDT         |
| ranking_score       | AF3 ranking score       |

| ptm                 | Global pTM              |
| fraction_disordered | Fraction disordered     |

Columns:

| Column              | Description            |
| ------------------- | ---------------------- |

| source_file         | Confidence JSON source |
| n_chains            | Number of chains       |
| n_pairs             | Number of chain pairs  |

| iptm                | Global ipTM            |

# Phase 6 — Size Correction

## Input

### pair_scores_raw.tsv

---

## Outputs

### pair_scores_size_corrected_observations.tsv

Size-corrected interaction observations.

Each row still represents a single observation.

Additional columns include:

| Column               | Description                 |
| -------------------- | --------------------------- |
| protein_1            | Canonicalized protein ID    |
| protein_2            | Canonicalized protein ID    |
| pair_tokens          | Combined protein length     |
| size_feature         | Length transformation       |
| size_baseline        | Expected score inflation    |
| score_size_corrected | Corrected interaction score |

---

### size_correction_model.tsv

Stores the fitted size-correction model.

Columns:

| Column          | Description                        |
| --------------- | ---------------------------------- |
| method          | Correction method                  |
| score_column    | Corrected score                    |
| root_to_use     | Root transformation                |
| correction_type | Additive or subtractive correction |
| intercept       | Model intercept                    |
| slope           | Model slope                        |
| formula         | Regression formula                 |
| n_observations  | Number of observations             |
| n_unique_pairs  | Number of unique interactions      |

Example formula:

```text
chain_pair_iptm ~ intercept +
slope * (protein_1_length + protein_2_length)^0.5
```

This file ensures full reproducibility of the correction procedure.

# Final Output

The main result of the workflow is:

```text
pair_scores_size_corrected_aggregated.tsv
```

Typical downstream applications include:

* Interaction ranking
* Candidate prioritization
* Network construction

* Biological interpretation

# Phase 8 — Bait Analysis

## Input

### pair_scores_size_corrected_aggregated.tsv

Final aggregated interaction table generated after size correction and replicate aggregation.

This file is used to extract all predicted interactions involving one protein of interest.

The protein of interest can appear in either:

* `protein_1`
* `protein_2`

---

## Outputs

Bait analysis outputs are saved in:

```text
project_output/
└── project_name/
    └── bait_analysis/
```

---

### filtered_<protein_of_interest>_<sort_column>.tsv

Bait-specific ranked interaction table.

This file contains only interactions involving the selected protein of interest.

Example:

```text
filtered_RFA1_raw_score_resolved.tsv
```

Columns are inherited from:

```text
pair_scores_size_corrected_aggregated.tsv
```

The table is sorted by the score column selected in the configuration file, for example:

```text
raw_score_resolved
```

This file is useful for:

* Identifying top predicted interactors
* Prioritizing candidates for downstream validation
* Inspecting all interactions involving a specific bait protein

---

### ranked_interaction_scores_<protein_of_interest>.svg

Ranked score plot for the selected protein of interest.

Example:

```text
ranked_interaction_scores_RFA1.svg
```

This plot shows the top-ranked interaction partners for the bait protein.

It is useful for quickly identifying:

* Highest-scoring predicted interactors
* Score separation between top candidates
* Candidate ranking patterns

---

### score_distribution_<score_column>.svg

Distribution plot of the selected interaction score.

Example:

```text
score_distribution_raw_score_resolved.svg
```

This plot shows the distribution of scores among all interactions involving the selected protein of interest.

It is useful for:

* Assessing whether a few interactions score much higher than the rest
* Evaluating score spread
* Choosing candidate cutoffs for downstream analysis

---

# Final Bait-Specific Output

The main bait-analysis result is:

```text
bait_analysis/filtered_<protein_of_interest>_<sort_column>.tsv
```

This file is the recommended output for bait-specific downstream analysis and candidate prioritization.
